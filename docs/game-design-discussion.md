# 遊戲設計討論（Kickoff）

這份文件用來快速對齊第一版玩法，避免一開始做太多。

## 先決策這 4 件事

1. **核心任務是什麼？**
2. **每局多長？**
3. **難度如何調整？**
4. **怎麼讓玩家願意每天回來？**

---

## 方案 A：科學優先（推薦先做）

- 任務：2AFC 左傾 / 右傾判斷
- 刺激：固定空間頻率（例如 4 cpd），先只調整對比
- 難度：2-down-1-up staircase
- 每局：3 blocks × 20 trials（約 5–8 分鐘）
- 優點：
  - 訊號乾淨，容易觀察 threshold 改變
  - 實作風險低，最快落地
- 缺點：
  - 遊戲感較弱，前期可能偏枯燥

## 方案 B：遊戲化優先

- 任務：保留方向判斷，但加入連擊、生命值、關卡主題
- 刺激：每關改背景/節奏，但核心刺激保持一致
- 難度：staircase + 關卡速度加成
- 每局：2–4 分鐘（更短）
- 優點：
  - 留存可能更好
- 缺點：
  - 混入太多變因，訓練指標雜訊變高

## 方案 C：平衡版（科學核心 + 輕遊戲化）

- 任務：與方案 A 相同
- 遊戲層：只加「每日任務 + 連續簽到 + 個人最佳」
- 每局：5 分鐘上下
- 優點：
  - 兼顧可測量性與可玩性
- 缺點：
  - 設計工作略高於方案 A

---

## 我建議的 MVP（v0.1）

先採 **方案 A**，但保留兩個輕量遊戲化元素：

1. **連續答對提示（streak feedback）**
2. **訓練後趨勢圖（今天 vs 7 日平均）**

這樣可以先驗證訓練有效性與資料穩定度，再迭代美術與任務包裝。

## 個人紀錄方案（你上一題的延伸）

- MVP：LocalStorage（簡單、快速）
- 若記錄量變大：升級 IndexedDB
- 不建議 cookie：
  - 容量小
  - 每次 HTTP 都會帶上，非必要
  - 不適合存大量 session/trial 歷史
